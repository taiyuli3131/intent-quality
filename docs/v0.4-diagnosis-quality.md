# v0.4 Diagnosis Quality

> Status: P0 implementation notes
> Date: 2026-06-22

## Goal

v0.4 P0 adds a read-only diagnosis calibration layer.

The goal is to make diagnosis quality easier to inspect before any generated case, eval, profile, rule, contribution, or public-sample candidate is treated as usable.

The calibration layer is local-first, file-based, confirmation-gated, and check-only. It validates synthetic calibration fixtures; it does not generate or approve real fixtures.

## Boundaries

Included:

- schema checks for diagnosis calibration fixtures;
- readiness classification as `ready`, `needs_review`, or `blocked`;
- required field and finding coverage checks;
- independent coverage matrix reporting for the P0 beta scope;
- structural comparison of fields, dimensions, status, confidence ranges, and gate shape;
- privacy and redaction blockers;
- confidence range checks;
- required, optional, and forbidden finding structure;
- premise status checks;
- completion-question checks for incomplete cases;
- authorization-scope checks for protected targets;
- candidate gate blockers for auto-apply, writes, or confirmation bypass;
- protected snapshot checks during `intent-quality check`.

Not included:

- hosted platform or dashboard;
- default LLM-as-judge scoring;
- complete semantic evaluation claims;
- real fixture generation or approval;
- automatic candidate adoption;
- feedback writing;
- real fixture adoption into public examples;
- profile, rule, dataset, casebook, rubric, contribution, public state, suggestion, candidate, or accepted-asset mutation.

## Schema

Diagnosis calibration fixtures use `schema_version: 0.4.0-alpha`.

Required top-level fields:

- `fixture_id`
- `title`
- `input`
- `expected_diagnosis`
- `calibration_notes`
- `diagnosis_quality_gate`

The manifest supplies each fixture's `expected_status`: `ready`, `needs_review`, or `blocked`.

`expected_diagnosis` contains:

- `required_findings`;
- `optional_findings`;
- `forbidden_findings`;
- `premise_status`;
- `required_completion_questions`;
- `expected_authorization_scope`;
- `learning_feedback_expectations`.

Required and optional findings use `confidence_range.min` and `confidence_range.max`. They must not use single-value confidence fields such as `confidence` or `confidence_level`.

`diagnosis_quality_gate.readiness` must be one of:

- `ready`: enough evidence exists for the diagnosis quality gate;
- `needs-human-review`: the fixture is structurally valid but needs manual completion;
- `blocked`: privacy, redaction, authorization, schema, or candidate-gate blockers prevent reuse.

The checker normalizes `needs-human-review` to the manifest status `needs_review`.

`diagnosis_quality_gate` also records:

- `case_candidate_signal.eligible`, `reasons`, and `blockers`;
- `eval_candidate_signal.eligible`, `reasons`, `blockers`, and `eval_dimensions`;
- `auto_apply_allowed: false`;
- `requires_confirmation: true`.

## Public Fixture Boundary

Public diagnosis calibration fixtures must be synthetic, redacted, or derived examples. Real collaboration samples are private local material by default. They should remain local and uncommitted unless the user explicitly approves a separate generalized contribution flow.

## Manual Completion Prompt

When a calibration fixture is `needs-human-review`, it must include targeted completion questions under `expected_diagnosis.required_completion_questions`.

Use questions that change the diagnosis outcome, such as:

- What exact user wording established the expected response mode?
- What source excerpt verifies or refutes the decision-critical premise?
- Did the user explicitly authorize file writes or durable state changes?
- Was there earlier context that allowed the Agent to change modes?

Questions should target response mode, authorization scope, premise validation, evidence, or expected versus actual behavior.

## Confidence Calibration

Confidence uses a numeric `confidence_range`:

- high-support findings should have a narrow high range backed by direct evidence;
- partial or inferred findings should use lower or wider ranges;
- unknown premise cases should normally remain `needs-human-review` unless the blocker itself is directly observable.

Ranges must stay between `0` and `1`, and `min` must be less than or equal to `max`.

## Candidate Quality Gate

The calibration gate remains preview-only or blocked. A candidate signal must not:

- auto-apply;
- write local assets;
- bypass confirmation;
- allow adoption without user confirmation.

Any fixture that represents automatic application, local asset writes, confirmation bypass, or adoption without user confirmation should be blocked through `diagnosis_quality_gate.*_candidate_signal.blockers`.

## Beta Coverage Matrix

The P0 beta checker reports an independent coverage matrix for:

- authorization boundary;
- premature execution;
- premise validation;
- context pollution;
- response-mode mismatch;
- goal/intent preservation;
- confidence overstatement;
- forbidden inference or motive attribution;
- missing completion questions;
- blocked candidate gates.

This matrix is report-layer structural coverage only. It checks that fixture fields, dimensions, statuses, `confidence_range`, and gate signals are represented. It does not judge semantic correctness, does not use LLM-as-judge, and does not create replacement fixtures when coverage is missing.

Each coverage row reports:

- `covered: true | false`;
- `fixture_ids`;
- `status_mix` counts for `ready`, `needs_review`, and `blocked`;
- `approved_sample_count`, which counts only `ready` fixtures.

Blocked fixtures can count as blocked coverage for a dimension, but they are not approved samples and must not be treated as accepted baseline material.

The `check` command prints a diagnosis calibration gate report with:

- fixture counts;
- status counts;
- legacy coverage booleans for the original subset;
- the independent `coverage_matrix`;
- case/eval eligibility counts;
- reason and blocker counts;
- `requires_confirmation`;
- `auto_apply_allowed: false`.

## Acceptance

Alpha acceptance:

- calibration fixtures exist for ready, needs-review, privacy-blocked, and candidate-gate-blocked cases;
- `python -m intent_quality.cli check` validates the fixtures read-only;
- protected snapshots confirm checks do not mutate local assets.

Beta acceptance:

- additional diagnosis calibration fixtures cover context pollution, response-mode mismatch, intent preservation, missing completion questions, and forbidden inference/confidence overstatement;
- check output includes the diagnosis calibration gate report and independent coverage matrix;
- documentation and fixture examples remain consistent with schema behavior;
- all blockers produce inspectable reasons without claiming semantic completeness.

Candidate acceptance:

- calibration checks remain local-only and read-only;
- no fixture, candidate, feedback, profile, rule, dataset, casebook, rubric, contribution, public state, or accepted local asset is generated, adopted, or modified by `check`;
- the project continues to describe diagnosis calibration as quality gating, not as a complete semantic evaluator.
