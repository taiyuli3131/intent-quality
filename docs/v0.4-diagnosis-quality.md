# v0.4 Diagnosis Quality

> Status: P0 implementation notes
> Date: 2026-06-22

## Goal

v0.4 P0 adds a read-only diagnosis calibration layer.

The goal is to make diagnosis quality easier to inspect before any generated case, eval, profile, rule, contribution, or public-sample candidate is treated as usable.

The calibration layer is local-first, file-based, confirmation-gated, and check-only. It validates synthetic calibration fixtures; it does not generate or approve real fixtures.

## Boundaries

Included:

- schema checks for diagnosis calibration fixtures;
- readiness classification as `ready`, `needs_review`, or `blocked`;
- required field and finding coverage checks;
- privacy and redaction blockers;
- confidence range checks;
- required, optional, and forbidden finding structure;
- premise status checks;
- completion-question checks for incomplete cases;
- authorization-scope checks for protected targets;
- candidate gate blockers for auto-apply, writes, or confirmation bypass;
- protected snapshot checks during `intent-quality check`.

Not included:

- hosted platform or dashboard;
- default LLM-as-judge scoring;
- complete semantic evaluation claims;
- real fixture generation or approval;
- automatic candidate adoption;
- feedback writing;
- profile, rule, dataset, casebook, rubric, contribution, public state, suggestion, candidate, or accepted-asset mutation.

## Schema

Diagnosis calibration fixtures use `schema_version: 0.4.0-alpha`.

Required top-level fields:

- `fixture_id`
- `title`
- `expected_status`
- `source`
- `readiness`
- `authorization_scope`
- `confidence_calibration`
- `finding_requirements`
- `findings`
- `premises`
- `completion_questions`
- `privacy`
- `generated_candidates`
- `safety`

`expected_status` and `readiness.status` must match one of:

- `ready`: enough evidence exists for the diagnosis quality gate;
- `needs_review`: the fixture is structurally valid but needs manual completion;
- `blocked`: privacy, redaction, authorization, schema, or candidate-gate blockers prevent reuse.

## Manual Completion Prompt

When a calibration fixture is `needs_review`, it must include targeted completion questions.

Use questions that change the diagnosis outcome, such as:

- What exact user wording established the expected response mode?
- What source excerpt verifies or refutes the decision-critical premise?
- Did the user explicitly authorize file writes or durable state changes?
- Was there earlier context that allowed the Agent to change modes?

Questions should target `response_mode`, `authorization_scope`, `premise_validation`, `evidence`, or `expected_vs_actual_mode`.

## Confidence Calibration

Confidence uses a label plus a numeric score:

- `high`: strong direct evidence from user wording and observed Agent behavior;
- `medium`: enough evidence for a likely diagnosis, but some interpretation remains;
- `low`: manual summary, missing source evidence, unknown premise status, or incomplete action evidence.

`confidence_calibration.score` must be between `0` and `1`.

Low-confidence fixtures can still be useful, but they should normally be `needs_review` unless the blocker itself is directly observable, such as a private path or candidate confirmation bypass.

## Candidate Quality Gate

Generated candidates remain preview-only or blocked. A candidate must not:

- auto-apply;
- write local assets;
- bypass confirmation;
- allow adoption without user confirmation.

Any candidate with `auto_apply: true`, `writes_local_asset: true`, `requires_user_confirmation: false`, or `gate.adoption_allowed_without_confirmation: true` is blocked by the candidate quality gate.

## Acceptance

Alpha acceptance:

- calibration fixtures exist for ready, needs-review, privacy-blocked, and candidate-gate-blocked cases;
- `python -m intent_quality.cli check` validates the fixtures read-only;
- protected snapshots confirm checks do not mutate local assets.

Beta acceptance:

- additional diagnosis calibration fixtures cover context pollution, response-mode mismatch, and intent preservation;
- documentation and fixture examples remain consistent with schema behavior;
- all blockers produce inspectable reasons without claiming semantic completeness.

Candidate acceptance:

- calibration checks remain local-only and read-only;
- no fixture, candidate, feedback, profile, rule, dataset, casebook, rubric, contribution, public state, or accepted local asset is generated, adopted, or modified by `check`;
- the project continues to describe diagnosis calibration as quality gating, not as a complete semantic evaluator.
