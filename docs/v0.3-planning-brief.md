# v0.3 Planning Brief

> Status: planning seed
> Created: 2026-06-18

## Current Baseline

`intent-quality` is currently accepted at `v0.2-candidate-accepted`.

v0.2 completed the reliability hardening layer:

- public sync gates now inspect real candidate content before suggestion generation;
- eval scorer output is more inspectable and regression-friendly;
- all meaningful mutations remain confirmation-gated;
- the project remains local-first and file-based.

## Planning Goal

v0.3 should be planned before implementation.

The main question is which direction creates the most value after v0.2:

- better diagnosis quality;
- better learning and playbook guidance;
- better local profile memory;
- adapter export;
- stronger eval review;
- public registry maturity.

v0.3 should not begin by adding platform features unless the product direction explicitly changes.

## Candidate Themes

### P0: Diagnosis Quality

Improve the `diagnose` path so it can guide incomplete manual inputs more precisely.

Possible scope:

- confidence levels per finding;
- targeted completion questions;
- stronger distinction between user-stated, inferred, assumed, and verified premises;
- clearer learning notes inside diagnosis reports;
- better case/eval candidate generation from diagnosis output.

### P1: Learning And Playbook

Expand user-facing learning support without turning the product into generic documentation.

Possible scope:

- context pollution;
- premise validation;
- response mode selection;
- diagnose versus eval;
- public sample trust;
- suggestion review;
- contribution privacy.

### P1: Local Profile Memory

Make profile memory more useful while staying confirmation-gated.

Possible scope:

- profile update suggestions;
- recurring failure pattern summaries;
- reviewable local memory entries;
- stale-memory warnings;
- profile rollback notes.

### P2: Adapter Export

Explore adapter-ready output as experimental/internal only.

Possible scope:

- Promptfoo config draft export;
- DeepEval test case draft export;
- Pydantic Evals dataset draft export.

Adapters should not become core runtime dependencies in v0.3 unless explicitly accepted.

### P2: Stronger Eval Review

Improve review paths without claiming full semantic evaluation.

Possible scope:

- human-review status workflow;
- clearer reviewer notes;
- optional LLM-as-judge spike behind an experimental flag;
- calibration notes for false positives and false negatives.

## Recommended v0.3 Non-Goals

- hosted accounts;
- full dashboard;
- automatic public upload;
- automatic public-sample adoption;
- automatic profile/rule/dataset/casebook/rubric/contribution mutation;
- production registry signatures;
- full semantic evaluator as the default scorer.
