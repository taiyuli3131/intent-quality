# intent-quality v0.3 Roadmap

> Status: candidate consistency cleanup
> Date: 2026-06-22
> Baseline tag: `v0.3-open-source-foundation`

## Summary

v0.3 should make `diagnose` trustworthy enough to become the main product entry point.

v0.2 hardened the local trust layer around public sync and eval scoring. v0.3 should return to the user-facing loop: diagnosing real collaboration issues, explaining what happened, guiding missing information, teaching the relevant concept, and creating reviewable local candidates without applying them automatically.

v0.3 is not a platformization release. It keeps the local-first, file-based, confirmation-gated model.

Current local v0.3 baselines now cover diagnosis quality, playbook learning, local profile memory, eval review, adapter export, and open-source foundation readiness. Candidate cleanup should keep these workstreams consistent rather than expanding scope.

## Product Goal

The v0.3 goal is:

```text
Improve diagnosis quality and learning conversion while preserving the v0.2 safety model.
```

The release should help users answer:

- what happened in this Agent interaction;
- which facts are known, user-stated, inferred, assumed, verified, or unknown;
- what evidence supports each finding;
- what missing information would improve the diagnosis;
- what concept the user should learn from the issue;
- which case, eval, profile, rule, playbook, or contribution candidates are worth reviewing.

## Priorities

### P0: Diagnosis Quality

Diagnosis quality is the main v0.3 product priority.

Included:

- finding-level confidence and evidence;
- evidence source and evidence type in diagnosis output;
- targeted completion questions for incomplete manual and conversation inputs;
- clearer distinction between `user-stated`, `inferred`, `assumed`, `verified`, and `unknown` premises;
- stronger expected-mode versus actual-mode analysis;
- authorization-scope analysis for file writes, profile/rule changes, dataset/casebook changes, rubric changes, contribution actions, and public-sample adoption;
- clearer learning notes inside diagnosis reports;
- better preview-only case and eval candidate generation from diagnosis output.

P0 should not create accepted casebook entries, accepted datasets, applied profile changes, applied rule changes, or submitted contributions automatically.

### P1: Learning And Playbook

Expand the learning layer enough to support the concepts users see in diagnosis reports.

Included:

- playbook pages for context pollution;
- premise validation;
- response mode;
- diagnose versus eval;
- public sample trust;
- suggestions and confirmation;
- contribution privacy;
- diagnosis report links from findings to related concepts.

The playbook should remain a compact local learning layer, not a generic AI course or documentation platform.

### P1: Local Profile Memory

Local profile memory may enter v0.3 only as reviewable suggestions.

Included:

- profile update suggestions derived from diagnosis evidence;
- recurring failure pattern summaries;
- stale-memory warnings;
- reviewable local memory entries;
- impact scope and rollback notes for any proposed profile change.

Profile memory remains limited to Agent collaboration behavior. It must not become broad personality storage, default cross-project memory, or automatic profile mutation.

### P1: Eval Review

Add a stronger review path for ambiguous eval outcomes without changing the default scorer's claim.

Included:

- human-review status workflow;
- reviewer notes;
- review decisions for `needs_review` outputs;
- false-positive and false-negative calibration notes.

Current v0.3 baseline records this as local human-review metadata for `needs_review` outputs. It does not change the default heuristic scorer, apply eval results, mutate datasets or rubrics, or claim complete semantic evaluation.

An optional LLM-as-judge spike is deferred unless explicitly scoped later. It must not become the default scorer or be described as proof of semantic quality.

### P2: Adapter Export

Adapter export may be explored as experimental/internal output only.

Possible draft exports:

- Promptfoo config draft;
- DeepEval test case draft;
- Pydantic Evals dataset draft.

Adapter exports should include source metadata, schema/rubric versions, scorer limitations, and experimental status. They should not become core runtime dependencies in v0.3.

Current v0.3 baseline supports draft exports only. The exports do not run external frameworks, add dependencies, replace the heuristic scorer, apply results, or mutate local assets.

### P2: Public Registry

Public registry work should stay below platform scope in v0.3.

Allowed work:

- fixture maintenance;
- documentation consistency;
- relevance explanation improvements;
- additional read-only checks for public sample safety.

Deferred work:

- production registry signatures;
- hosted registry governance;
- automatic public upload;
- automatic public-sample adoption;
- public contribution submission service.

## In Scope

v0.3 includes:

- diagnosis report quality improvements;
- diagnosis YAML/schema extensions where needed for evidence, confidence, premise status, and completion questions;
- preview-only diagnosis-derived case and eval candidates;
- playbook concept coverage for diagnosis feedback;
- confirmation-gated local profile suggestions;
- human-review records for eval review;
- experimental/internal adapter export drafts;
- read-only regression checks for the new planning scope.

## Deferred Scope

Deferred beyond v0.3:

- full dashboard;
- hosted accounts;
- enterprise governance workflows;
- production public registry trust infrastructure;
- automatic public upload;
- automatic public-sample adoption;
- default LLM-as-judge scoring;
- complete semantic evaluation;
- external eval frameworks as core runtime dependencies;
- broad global memory or cross-project history analysis;
- apply/dismiss/edit suggestion workflows beyond the reviewable local proposal model unless explicitly scoped later.

## Non-Goals

v0.3 must not:

- expand the product into a hosted platform;
- change the local-first principle;
- automatically apply profile, rule, dataset, casebook, rubric, contribution, or public sample changes;
- present public samples as trusted before local checks and user acceptance;
- treat contribution package generation as submission authorization;
- claim the heuristic scorer is a complete semantic evaluator;
- make adapter integrations core runtime dependencies;
- run external eval frameworks as part of the default local loop;
- use local profile memory as a broad personal memory system.

## Acceptance Criteria

v0.3 can be considered candidate-ready when:

- `diagnose` produces finding-level evidence, confidence, and missing-information questions for manual and conversation inputs;
- diagnosis output separates facts, user-stated claims, inferred conclusions, assumptions, verified premises, and unknowns;
- expected mode, actual mode, and authorization scope are visible in the Markdown and YAML report;
- generated case, eval, profile, rule, contribution, and public-sample actions remain preview-only or pending suggestions;
- playbook pages cover the diagnosis concepts used in reports;
- local profile memory outputs are pending suggestions with source evidence, impact scope, confirmation state, and rollback notes;
- eval review can record human-review status and reviewer notes for `needs_review` outputs;
- any LLM-as-judge work is explicitly experimental and does not replace the heuristic scorer;
- adapter exports are clearly marked experimental/internal and do not add core runtime dependencies;
- `intent-quality check` remains read-only and covers public sync fixtures, eval response fixtures, eval review fixtures, diagnosis quality fixtures, profile memory fixtures, adapter export fixtures, and playbook links;
- read-only checks confirm no automatic mutation of profiles, rules, datasets, casebooks, rubrics, suggestions, candidates, contributions, or submissions;
- documentation continues to state the local-first safety model and scorer limitations.
